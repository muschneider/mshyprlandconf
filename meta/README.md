# `meta/` — vendored Hyprland metadata

This directory holds the **upstream source of truth** that `hyprconf-core`'s
schema is derived from and checked against. Nothing here is edited by hand.

## Files

| File                       | Source                                                  | Purpose                                                                 |
| -------------------------- | ------------------------------------------------------- | ----------------------------------------------------------------------- |
| `hl.meta.lua`              | `/usr/share/hypr/stubs/hl.meta.lua`                     | The official autogenerated Lua API stub shipped by Hyprland.            |
| `hyprland-config-keys.txt` | Extracted from `hl.meta.lua` (`---@alias HL.ConfigKey`) | The authoritative, flat list of every configuration option key.         |

### Provenance

- Hyprland **0.55.2** (`v0.55.2`, commit `39d7e209…`, tagged 2026-05-16),
  as reported by `hyprctl version` on the build host.
- `hl.meta.lua` header: _"This file is autogenerated. Do not edit by hand.
  Generator: scripts/generateLuaStubs.py"_.
- `hyprland-config-keys.txt` is the 341-entry `HL.ConfigKey` union from that
  stub, one `section.key` per line (dotted form; e.g. `decoration.blur.size`,
  `general.col.active_border`).

## How the schema maps to these keys

`hyprconf-core` stores option paths in **hyprlang `conf` notation**, where
sub-sections are separated by `:` and dotted key names (like `col.active_border`)
keep their `.`:

| `conf` path (schema)            | stub key (`hl.meta.lua`)        |
| ------------------------------- | ------------------------------- |
| `decoration:rounding`           | `decoration.rounding`           |
| `decoration:blur:size`          | `decoration.blur.size`          |
| `general:col.active_border`     | `general.col.active_border`     |
| `group:groupbar:col.active`     | `group.groupbar.col.active`     |
| `input:touchpad:natural_scroll` | `input.touchpad.natural_scroll` |

The mapping is mechanical: **replace every `:` with `.`**. The unit test
`schema::tests::every_option_path_exists_in_vendored_stub` performs exactly this
transform for every `OptionSpec` in the schema and asserts membership in
`hyprland-config-keys.txt`, so the schema can never reference an option that the
shipped Hyprland does not know about.

## Regenerating after a Hyprland upgrade

```sh
# 1. Refresh the vendored stub.
cp /usr/share/hypr/stubs/hl.meta.lua meta/hl.meta.lua

# 2. Re-extract the flat key list (drops the trailing blank line).
sed -n '/^---@alias HL.ConfigKey/,/^[^-]/p' meta/hl.meta.lua \
  | sed -n 's/^---| "\(.*\)"$/\1/p' \
  | grep '\.' > meta/hyprland-config-keys.txt

# 3. Run the tests; coverage gaps and stale keys surface as failures.
just test
```

## What is NOT here (yet)

The stub enumerates *keys* but not their value **types**, **defaults**, ranges,
or human descriptions. Those are sourced from the
[Hyprland wiki — Configuring/Variables](https://wiki.hyprland.org/Configuring/Variables/)
and encoded by hand in `crates/hyprconf-core/src/schema/data.rs`. Defaults in
particular drift between releases; verify against `hyprctl getoption <name>` on
the target version when accuracy matters.
